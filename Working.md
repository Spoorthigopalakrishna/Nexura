# Nexura Architecture & Technical Documentation

This document provides a comprehensive overview of the Nexura B2B Procurement Hub's internal workings, data flow, and architectural principles.

## 🏛 System Architecture

Nexura is built as a modular React application using a **Contextual Data Architecture**. Instead of complex state libraries, it utilizes the React Context API to manage distinct domains of the procurement lifecycle.

### Core Modules
1.  **Auth Module (`AuthContext`)**: Handles role-based identity. Roles include `Employee`, `Manager`, `Finance`, and `Admin`. Navigation and permissions are dynamically derived from the authenticated role.
2.  **Product Module (`CatalogPage`)**: A high-performance discovery interface with real-time filtering and local asset management for enterprise products.
3.  **Requisition Module (`CartContext` & `RequisitionContext`)**: Manages the transition from a shopping cart to a formal Purchase Requisition (PR). 
4.  **Approval Engine**: A rules-driven sequential workflow that routes requests based on organizational criteria.

## 🔄 Core Workflows

### 1. Discovery & Cart
Items are searched and filtered in the `CatalogPage`. When added to the cart, they are managed by `CartContext`. The cart calculates real-time totals in INR (₹) and maintains persistence during the session.

### 2. Requisition Submission
When an employee submits a requisition, the `RequisitionPage` captures business justification and priority. This data is passed to the `RequisitionContext`, which triggers the **Rules Engine**.

### 3. Automated Rules Engine
The engine analyze the request based on two primary dimensions:
- **Total Amount**:
  - `> ₹50,000`: Requires Finance approval.
  - `> ₹2,00,000`: Requires Admin approval.
- **Priority**:
  - `Critical`: Triggers expedited routing to the highest available level.

The engine generates an `approvalChain`—an array of `ApprovalStep` objects representing the required sign-offs.

### 4. Sequential Approval Flow
Requisitions are routed through the `approvalChain` index by index. 
- Only the user with the role matching `currentStepIndex` can see the requisition in their **Approvals Dashboard**.
- **Approvals** increment the index and move the status to `Pending_Approval` for the next level.
- **Rejections** terminate the chain immediately and set the status to `Rejected`.

## 🎨 UI & Design Principles

Nexura utilizes a custom-built **Glassmorphic Design System** defined in `src/styles/index.css`.

- **Aesthetics**: Dark-mode primary theme with cyan and purple accents.
- **Components**: Utilizes `lucide-react` for industrial iconography and `framer-motion` (planned for Phase 3) for micro-animations.
- **Responsiveness**: Flexbox and CSS Grid are used for fluid layouts that adapt to various terminal and monitor sizes.

## 📊 Data Models (TypeScript)

### Requisition Statuses
- `Draft`: Initial state (in-cart).
- `Pending_Approval`: Currently in the approval chain.
- `Approved`: Final authorization granted.
- `Rejected`: Denied by an approver.
- `Ordered`: Sent to supplier (Phase 4).

### Key Interfaces
```typescript
interface Requisition {
  id: string;            // Standard PR format (PR-XXXX)
  requestor: User;       // User details including department
  items: CartItem[];     // Snapshot of products at submission
  totalAmount: number;   // Calculated sum
  approvalChain: Step[]; // Generated by the Rules Engine
  status: Status;        // Current state in lifecycle
}
```
